# 三个编辑系统的集成评估

## 结论

适合采用渐进式集成，不建议现在把三个项目直接揉成一个大 Flask 文件。

当前三个项目的后端并不一致：编辑部数据汇总的 AI 查询工具使用 Python 自带的 `ThreadingHTTPServer`，伦理审查和引用检查使用 Flask。两个 Flask 项目还各自维护上传、下载、长任务、历史记录与临时目录。如果直接合并路由和全局变量，容易出现配置模块重名、`/api/settings` 路由冲突、工作目录错位和任务状态串扰。

本次实现采用四层结构：

1. 统一工作台：用户只访问一个网页。
2. 模块隔离：三个副本分别在独立本机端口运行，保留成熟逻辑。
3. 同源代理：工作台以 `/modules/data/`、`/modules/ethics/`、`/modules/citation/` 暴露模块，页面内切换不会跳到不同地址。
4. 共享配置：三者读取同一份本机 AI 配置。

## 不只是导航栏的部分

统一层承担了这些实际能力：

- 一个启动入口同时管理三个服务的生命周期。
- 首页直接显示稿件数、作者记录数、编辑部数和 ETL 提示数。
- 显示三个模块的运行状态和 AI 配置状态。
- 模块页面使用保留状态的内嵌工作区，来回切换不会清空已输入内容。
- 设置窗口统一管理 API Key、接口地址、模型和工作台主题。
- 数据库重建成为工作台级操作，直接使用当前工作区中的 Excel 与映射配置。
- 模块可在工作台内使用，也可从工具条打开独立窗口。

## 文件复制策略

### 已复制

- 三个项目的业务源码、HTML 页面、依赖清单和说明。
- 数据汇总的当前 SQLite 数据库。
- 数据汇总的映射 CSV 与当前原始 Excel。
- 当前科研诚信风险报告。
- 数据同步脚本本身。
- 引用检查中已配置的 AI 设置，迁移到工作区本机配置。

### 未复制

- 三个项目的 `.git` 历史。
- `build`、`dist`、`__pycache__`、Playwright 产物和临时任务目录。
- 数据库历史备份，只保留当前数据库。
- Google `credentials.json` 与 `token.pickle`。

这些排除项不是系统运行必需文件，且构建产物、历史备份与凭据会放大体积或带来不必要的安全风险。

## 统一 AI 配置

统一配置沿用稿件引用检查的本机 JSON 方案，并增强为：

- Key 留空时保留现有值。
- 保存前测试连接。
- 支持只测试、不保存。
- 页面返回掩码，不回传明文 Key。
- 使用临时文件替换，降低写入中断导致配置损坏的风险。
- 三个模块每次任务读取共享文件，新设置无需重启即可生效。

DeepSeek 默认地址统一为完整 Chat Completions 地址：

```text
https://api.deepseek.com/v1/chat/completions
```

## 下一阶段建议

当前架构先解决“一个入口、一套设置、顺畅切换和独立运行”的问题。下一阶段如果要继续深化，建议按以下顺序推进：

1. 建立统一稿件标识，把期刊代码和稿件号作为跨模块主键。
2. 将伦理审查结果和引用检查结果写入独立结果库，并在数据汇总页展示综合风险。
3. 抽取共用的任务中心，统一查看处理中、已完成、失败和报告下载。
4. 把 OJS 地址、超时、工作目录等非 AI 配置也逐步放入统一设置。
5. 最后再考虑合并前端组件，不急于合并业务内核。

这样可以持续获得集成收益，同时避免一次性重写导致现有工具不可用。
